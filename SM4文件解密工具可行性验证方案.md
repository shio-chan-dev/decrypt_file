# SM4 文件/字符串解密工具可行性验证方案

## 一、目标

验证当前需求是否可以落地：

1. 支持国密 SM4 解密。
2. 支持字符串密文解密。
3. 支持文件流解密。
4. 已知密钥、IV、密文或加密文件。
5. 验证 RTX 4090/4070/4060 下的 GPU 加速效果，并与 CPU 解密结果进行对比。

## 二、AI 给出的脚本用途和待验证点

目标功能大致包括：

1. 使用 SM4 算法进行解密。
2. 支持字符串密文解密。
3. 支持文件流式解密。
4. 接收已知密钥、IV 和密文数据作为输入。
5. 在程序启动时检测当前机器是否存在 NVIDIA GPU。

待补充验证点：

1. 当前脚本中仍有占位内容，需要补充真实测试输入后再运行验证。
2. 代码已经包含 GPU 检测逻辑，SM4 运算是否实际在 GPU 上执行需要进一步确认。
3. 实际解密逻辑依赖 `cryptography`，更接近 CPU 解密方案。
4. 当前缺少真实密文样例、文件样例、正确性验证和性能测试。
5. RTX 4090/4070/4060 对 SM4 解密性能的提升效果，需要通过 CPU/GPU 对比测试确认。

## 三、需要先确认的信息

1. SM4 使用模式：CBC 还是 CTR。
2. 如果是 CBC，确认 padding 方式是否为 PKCS7。
3. 密文格式：原始 bytes、hex、base64，还是文件二进制。
4. IV 来源：固定提供，还是写在密文/文件头里。
5. 目标文件大小：MB 级、GB 级，还是 100GB 以上。
6. 性能目标：例如每秒解密多少 MB/GB。

## 四、验证步骤

### 1. 正确性验证

使用已知 key、IV、明文、密文样例，先实现 CPU 版 SM4 解密。

验证字符串解密结果是否和预期明文完全一致。

### 2. 文件解密验证

准备一个小文件和一个较大文件，使用相同 SM4 参数加密后再解密。

验证解密后的文件和原始文件完全一致。

### 3. 性能验证

先分别测试不同大小文件的 CPU 解密速度，建立性能基线，例如：

1. 100MB
2. 1GB
3. 10GB

记录耗时、吞吐量、CPU 占用和磁盘 IO 情况。

### 4. CPU/GPU 对比验证

在相同密钥、IV、加密模式和测试文件下，分别执行 CPU 解密和 GPU 解密。

对比两种方案的解密耗时、吞吐量、CPU 占用、GPU 占用和磁盘 IO 情况。

通过实测数据判断 GPU 方案相对 CPU 方案是否存在明显性能提升，以及提升主要来自 SM4 计算本身还是文件读写、数据传输等环节。

## 五、成功标准

1. 字符串密文可以稳定解出正确明文。
2. 文件解密后与原文件 hash 完全一致。
3. 错误 key、错误 IV、错误模式时能明确失败。
4. 有明确的 CPU/GPU 对比数据，包括耗时、吞吐量和资源占用。
5. 如果验证 GPU 加速效果，需要确认 SM4 运算实际发生在 GPU 上，而不只是检测到 GPU。


## 七、建议准备的验证材料

为便于客观验证方案效果，建议尽量准备以下测试材料。如果暂时无法提供完整材料，也可以先提供一组最小可验证样例，后续再补充文件样例和性能测试样例。

### 1. 一组可验证的样例

1. 明文。
2. 对应密文。
3. 测试用 SM4 密钥。
4. 测试用 IV 向量。
5. 加密模式：CBC 还是 CTR。
6. 如果是 CBC，建议确认 padding 是否为 PKCS7。

### 2. 字符串密文样例

1. 字符串密文格式：hex、base64，还是原始 bytes。
2. 预期解密后的字符串内容，便于核对解密结果。

### 3. 文件样例

1. 一个小的加密文件，用来验证功能是否正确。
2. 对应的原始文件，或者原始文件的 sha256 hash，便于核对文件解密结果。
3. 如方便提供，再准备一个接近真实业务大小的加密文件，用来做 CPU/GPU 性能对比。
